LiveKit Voice Agents — production-ready за час

Источник: worksh.app/tutorials/livekit-voice-agent Дата разбора: 20.07.2026 Уроков: 7 Стек: Python 3.10+ · livekit-agents 1.3

Конспект годного воркшопа по LiveKit Agents SDK: как за час собрать голосовой AI-агент, который не рубит клиента на полуслове, умеет откатываться на резервного провайдера, вызывает MCP-инструменты и корректно эскалирует на человека. Разложено по всем семи урокам с ключевыми абстракциями, минимальными кодовыми примерами и цифрами latency. Полезно тем, кто делает голосовые сценарии для контакт-центров и IVR-ботов, или собирает голосовой frontend поверх текстового агента.

TL;DR

Пайплайн: Voice Activity Detection → STT → LLM → TTS. LiveKit SDK даёт готовый оркестратор всех четырёх плюс WebRTC-транспорт, semantic turn detection, fallback между провайдерами, метрики и MCP-инструменты.

Целевая задержка (TTFA): ≤1000 мс. Складывается из VAD 15–30 мс + STT 200–600 мс + LLM 100–1000 мс + TTS 100–300 мс. Preemptive generation срезает 100–300 мс.

Ключевые фичи для контакт-центров: multilingual semantic turn detection (~20 мс, ловит незаконченные фразы), FallbackAdapter (auto-retry по цепочке провайдеров), MCP-серверы (внутренние сервисы можно прицепить без переписки), consent-задача с булевым исходом, эскалация к менеджеру / человеку по SIP.

Слабое место: preemptive generation ломает medical/legal/financial сценарии (модель начинает отвечать до полного ввода). Для compliance-чувствительных задач — не включать.

01 Introduction — production-ready voice agents

Setting the stage: почему голосовые агенты стало можно строить без audio-инженера.

Что изменилось за год

Год назад собрать голосовой агент = месяцы работы + специализированный audio-engineering бэкграунд. Сегодня — «working agent за несколько минут». Что упало в цене:

  • STT — быстрее и точнее (реалтайм с приемлемой WER для русского)
  • LLM — вытягивают нюансы диалога, а не только Q&A
  • TTS — звучит по-человечески, без «robotic voice»
  • LiveKit — абстрагирует WebRTC/оркестрацию, разработчик пишет только agent-логику

Что решают конкретно

Два боли, которые убивали UX голосовых ботов:

  • Robotic feel — VAD на порогах громкости даёт rigid turn-taking
  • «Cut you off» — агент отвечает на первой же паузе, до окончания мысли клиента

Практическое применение

Голосовые боты и copilot'ы для контакт-центров — прямой оверлеп с LiveKit. Ключевой вопрос — можно ли получить STT/TTS с русским качества, сопоставимого с AssemblyAI/Cartesia. По TTS кандидаты: Yandex SpeechKit, VseGPT, ElevenLabs Multilingual v2. По STT: Yandex, Deepgram Nova-2 (русский появился), собственный Whisper large-v3-turbo.

02 Foundations — пайплайн и первый agent.py

Почему WebRTC, а не HTTP/WS

Голосовой real-time — это media-задача. HTTP и WebSocket на её роль не проектировались. Ключевая разница — транспорт и что делаем при потере пакета.

Head-of-line blocking — главный убийца TCP для голоса

TCP гарантирует порядок и доставку. Пакет 42 потерялся — все пакеты 43+, уже пришедшие, лежат в буфере ОС и не отдаются приложению до тех пор, пока 42-й не повторно доедет. При потере 1% на мобильной сети это добавляет ~150–500 мс к задержке. Клиент слышит: агент говорит нормально, потом на секунду замолкает, потом «догоняет».

UDP отдаёт что пришло, в любом порядке. WebRTC поверх этого:

  • Пакет 42 пропал → PLC синтезирует ~20 мс правдоподобной паузы/интерполяции
  • Пакет 43 играется в свою слот-позицию
  • Слушатель слышит «щёлк» вместо «замолк на секунду»

Opus — почему именно этот кодек

Voice-агент выигрывает от Opus по нескольким пунктам:

  • Битрейт 6–510 кбит/с плавно подстраивается: на 4G голос идёт в ~24 кбит/с, на wifi может подняться до 64 кбит/с для музыкального качества (актуально для стриминга TTS с эмоциональной интонацией).
  • Frame size 2.5–60 мс — LiveKit обычно 20 мс. Чем меньше frame, тем ниже добавочная latency (но больше overhead-байтов на пакет).
  • FEC (Forward Error Correction) — в каждом пакете спрятана «тень» предыдущего. При потере 1–2 пакетов подряд декодер восстанавливает без искажений.
  • Hybrid SILK/CELT — переключается между речевым (SILK, узкополосный) и музыкальным (CELT, полнополосный) режимами в реальном времени. Для TTS с интонацией и фоновой музыкой (jingle) важно.
  • Роялти-free, поддержка везде (Chrome, Safari, iOS, Android, все SIP-шлюзы), стандарт IETF (RFC 6716).

Per-packet timestamps & jitter buffer

В сети пакеты приходят с переменной задержкой: пакет N+1 может опередить N. Без выравнивания получится «квакающий» звук.

RTP-header в каждом Opus-пакете содержит:

  • sequence number — увеличивается на 1 каждый пакет; позволяет обнаружить пропуск и переупорядочить
  • timestamp — момент когда сэмпл был захвачен на источнике (не когда отправлен); позволяет восстановить original timing
  • SSRC — идентификатор источника (важно для конференций, когда участников много)

Jitter buffer на приёме:

  • Накапливает пакеты 20–200 мс (адаптивно к нестабильности сети)
  • Пересобирает по sequence number в правильный порядок
  • Отдаёт декодеру ровным потоком, «замедляя» пакеты которые пришли слишком быстро и «дожидаясь» отстающих
  • Балансирует: маленький buffer = меньше latency, но больше глюков; большой = гладко, но клиент чувствует «задержку»

Adaptive Bitrate (ABR) через REMB / TWCC

Сеть меняется в течение звонка: клиент вышел из метро, зашёл в лифт, переключился с wifi на 4G. WebRTC мониторит packet loss + RTT и на лету пересобирает bitrate:

  • Приёмник шлёт отправителю RTCP-feedback (REMB — Receiver Estimated Maximum Bitrate; или новее TWCC — Transport-Wide Congestion Control)
  • Отправитель на основе этого снижает bitrate Opus'а — с 32 кбит/с падает до 12, звук становится «телефонным», но не пропадает
  • Сеть восстановилась → bitrate обратно ползёт вверх, качество возвращается

HTTP/WS так не умеет. Если TCP-connection упёрся в congestion window, приложение узнаёт об этом только когда retransmit-таймер выстрелит — уже поздно.

NAT-traversal (ICE / STUN / TURN)

Voice-звонок из мобильного приложения → сервер. Мобильный телефон за NAT провайдера, наш сервер тоже часто за корпоративным NAT. Как соединиться:

  • STUN — «эй сервер, скажи какой у меня внешний IP:port». Клиент+сервер узнают адреса друг друга и пробуют прямой UDP-flow. Работает в ~80% случаев.
  • TURN — если прямая связь не пробивается (симметричный NAT корпоративного файрвола), пускаем через relay-сервер. Дороже (весь трафик через нас), но всегда работает.
  • ICE — механизм который пробует STUN, TURN, host candidates в правильном порядке и выбирает лучший.

LiveKit Cloud даёт ICE-инфру из коробки; для self-hosted нужен свой TURN (coturn), обычно на публичном IP-е.

Что это значит на практике

Контакт-центр на мобильных клиентах — типичный кейс где HTTP/WS ломается. Плохая сота, метро, лифт, переход между wifi и LTE — пакеты теряются в кластерах, retransmit copolition убивает latency. WebRTC вытягивает где HTTP-based решение (типа наивного «WebSocket + Opus-сэмплы» без jitter buffer) сдаётся.

Что нельзя делать наивно: цеплять voice-агент через собственный WebSocket из FastAPI и стримить туда PCM. Придётся написать jitter buffer, PLC, FEC, ABR — всё то что WebRTC уже отдаёт бесплатно.

Self-hosted в регулируемом контуре: LiveKit Server ставится в закрытый контур (например, банковский), TURN на публичном IP через DMZ. Данные (audio + transcripts) наружу не уходят, шифрование SRTP на транспорте + DTLS для handshake.

agent.py — базовая сборка агента с STT/LLM/TTS/VAD и background voice cancellation:

from dotenv import load_dotenv
from livekit import agents
from livekit.agents import Agent, AgentServer, AgentSession, JobContext, room_io
from livekit.plugins import noise_cancellation, silero

load_dotenv()

class Assistant(Agent):
    def __init__(self) -> None:
        super().__init__(instructions="You are a helpful voice AI assistant.")

server = AgentServer()

@server.rtc_session()
async def entrypoint(ctx: JobContext):
    session = AgentSession(
        stt="assemblyai/universal-streaming:en",
        llm="openai/gpt-4.1-mini",
        tts="cartesia/sonic-3",
        vad=silero.VAD.load(),
    )
    await session.start(
        agent=Assistant(),
        room=ctx.room,
        room_options=room_io.RoomOptions(
            audio_input=room_io.AudioInputOptions(
                noise_cancellation=noise_cancellation.BVC(),
            ),
        ),
    )

if __name__ == "__main__":
    agents.cli.run_app(server)

Запуск локально в консоли:

uv run agent.py download-files
uv run agent.py console

03 Semantic turn detection

Preventing awkward interruptions. Ключевая фича LiveKit, которая отличает production-ready от «MVP на VAD».

Проблема

VAD триггерит по звуку. Клиент задумался в середине фразы — VAD решает «тишина, значит очередь ответа». Агент перебивает. Semantic turn detector решает по смыслу: «предложение закончено грамматически» — можно отвечать; «нет глагола / есть незавершённая интонация» — ждём.

Механика

  • Отдельная multilingual модель, идёт в поставке SDK
  • Работает поверх VAD, не заменяет его
  • Латентность ~20 мс — «negligible impact»
  • Учитывает pause-паттерны разных языков (в русском своя pausa после «ну», «типа»)

Как включить

from livekit.plugins.turn_detector.multilingual import MultilingualModel

session = AgentSession(
    stt="assemblyai/universal-streaming:en",
    llm="openai/gpt-4.1-mini",
    tts="cartesia/sonic-3",
    vad=silero.VAD.load(),
    turn_detection=MultilingualModel(),   # ← добавили строку
)

Установка: uv add "livekit-agents[turn-detector]"

Trade-off

Без semantic turn detection получаем два бага: агент перебивает и STT-точность падает (модель дообрабатывает уже начатую генерацию). Стоимость включения — 20 мс к TTFA. Стоит всегда.

04 Personality & Fallbacks

Designing resilient assistant behavior. Как научить агента говорить по-своему и переживать падения провайдеров.

Personality через instructions

Голосовой prompt ≠ текстовому. Правила:

  • Короткие фразы, мало скобок — звучит естественнее
  • ≤3 предложения в ответе — иначе latency и потеря внимания
  • Диалект/идиомы явно — UK vs US, RU vs bilingual
  • Stage directions в system prompt — управлять эмоцией через «say this with mild sarcasm»
class Assistant(Agent):
    def __init__(self) -> None:
        super().__init__(
            instructions=(
                "You are an upbeat, slightly sarcastic voice AI for tech support. "
                "Help the caller fix issues without rambling, "
                "and keep replies under 3 sentences."
            ),
        )

FallbackAdapter

Каждая стадия пайплайна оборачивается в цепочку провайдеров с приоритетом. Основной падает — auto-retry на следующего без разрыва сессии.

session = AgentSession(
    llm=llm.FallbackAdapter([
        inference.LLM(model="openai/gpt-4.1-mini"),
        inference.LLM(model="google/gemini-2.5-flash"),
    ]),
    stt=stt.FallbackAdapter([
        inference.STT.from_model_string("assemblyai/universal-streaming:en"),
        inference.STT.from_model_string("deepgram/nova-3"),
    ]),
    tts=tts.FallbackAdapter([
        inference.TTS.from_model_string("cartesia/sonic-3:9626c31c-bec5-4cca-baa8-f8ba9e84c8bc"),
        inference.TTS.from_model_string("inworld/inworld-tts-1"),
    ]),
    vad=vad,
    turn_detection=MultilingualModel(),
)

Тест fallback: подсунуть невалидное имя модели primary → в консоли увидеть цепочку попыток и переход на secondary.

Дырка в документации

LiveKit не публикует timeouts / retry counts для FallbackAdapter — «auto-retry in priority order» и всё. Для прод-контура с жёсткими SLA придётся смотреть код или задавать явно через кастомные политики. Отдельный TODO при внедрении.

05 Metrics, Deployment, Latency Optimization

Measuring performance and deploying to production. Что мерять, куда деплоить, чем разгонять.

Ключевые метрики

  • TTFA (Time to First Audio) — от окончания реплики клиента до первого слова агента. Целевое ≤1000 мс.
  • TTFT (Time to First Token) — задержка LLM. Изолирует, что тормозит: STT, модель или синтез.
  • Token usage — стоимость сессии.
  • Interruption rate — как часто клиент перебивает агента (сигнал, что агент болтливый).
  • Fallback activations — как часто primary провайдер валится.

Сбор метрик

from livekit.agents import AgentStateChangedEvent, MetricsCollectedEvent, metrics
import logging, time

logger = logging.getLogger(__name__)

# внутри entrypoint, до session.start(...)
usage_collector = metrics.UsageCollector()
last_eou_metrics: metrics.EOUMetrics | None = None

@session.on("metrics_collected")
def _on_metrics_collected(ev: MetricsCollectedEvent):
    nonlocal last_eou_metrics
    if ev.metrics.type == "eou_metrics":
        last_eou_metrics = ev.metrics
    metrics.log_metrics(ev.metrics)
    usage_collector.collect(ev.metrics)

async def log_usage():
    summary = usage_collector.get_summary()
    logger.info("Usage summary: %s", summary)

ctx.add_shutdown_callback(log_usage)

@session.on("agent_state_changed")
def _on_agent_state_changed(ev: AgentStateChangedEvent):
    if ev.new_state == "speaking":
        if last_eou_metrics:
            elapsed = time.time() - last_eou_metrics.timestamp
            logger.info(f"Time to first audio: {elapsed:.3f}s")

Жизненный цикл агента: listening → thinking → speaking. Метрики эмиттятся после каждой стадии.

Preemptive generation

Флаг, при котором LLM начинает генерировать ответ до окончания реплики клиента, но говорить начинает только после semantic turn end. Экономит 100–300 мс.

session = AgentSession(
    # ... остальной конфиг ...
    preemptive_generation=True,
)

Когда НЕ включать preemptive

Клиент может «переиграть» середину фразы. Модель уже сгенерировала ответ на первую часть — получаем ответ на неполный запрос. Категорически нельзя для medical / legal / financial и любых других compliance-чувствительных сценариев. Ок для tech-support-чата и всего conversational.

Общие правила разгона

  • Стримить всё: STT/LLM/TTS работают инкрементально, не батчами
  • Параллелить: TTS начинается на первых токенах LLM, а не после полного ответа
  • Async tool calls с bounded timeouts — блокирующие I/O выкидываем
  • Prompt tightness — короче prompt = быстрее TTFT и дешевле

06 Tools & MCP Integration

Connecting external systems safely. Как приделать агенту guarded-инструменты и внешние MCP-серверы.

@function_tool

Декоратор регистрирует Python-функцию как tool, доступный LLM. Ключевое: docstring становится description для LLM. Тип-хинты параметров — контракт входов.

import httpx
from livekit.agents import function_tool, RunContext, ToolError

class Assistant(Agent):
    @function_tool()
    async def lookup_weather(
        self,
        context: RunContext,
        location: str,
    ) -> dict:
        """Look up current weather for a location.

        Args:
            location: City name or location to get weather for.
        """
        async with httpx.AsyncClient() as client:
            geo = (await client.get(
                "https://geocoding-api.open-meteo.com/v1/search",
                params={"name": location, "count": 1}
            )).json()
            if not geo.get("results"):
                raise ToolError(f"Could not find location: {location}")
            lat, lon = geo["results"][0]["latitude"], geo["results"][0]["longitude"]
            weather = (await client.get(
                "https://api.open-meteo.com/v1/forecast",
                params={"latitude": lat, "longitude": lon,
                        "current": "temperature_2m,weather_code",
                        "temperature_unit": "fahrenheit"}
            )).json()
            return {
                "location": geo["results"][0]["name"],
                "temperature_f": weather["current"]["temperature_2m"],
                "conditions": weather["current"]["weather_code"],
            }

MCP-серверы напрямую

MCP цепляются в AgentSession — при старте SDK тянет список инструментов и регистрирует наряду с local @function_tool. Пример подключения официального LiveKit-docs MCP:

from livekit.agents import mcp

session = AgentSession(
    mcp_servers=[
        mcp.MCPServerHTTP(url="https://docs.livekit.io/mcp")
    ],
    # ... остальное ...
)

Установка: uv add "livekit-agents[mcp]". На практике: сюда логично цеплять внутренние текстовые сервисы компании через один общий MCP-роутер (типа mcp-proxy, sparfenyuk) — тогда voice-агент получает доступ к ним без переписи.

Безопасность в voice-контексте

  • Долгие операции — context.session.say("Одну секунду, ищу…") перед вызовом, чтобы клиент не думал что бот завис
  • Необратимые действия — context.disallow_interruptions() в начале tool-а (отправка заказа, списание, эскалация)
  • Ошибки — raise ToolError(...), LLM сам сформулирует пояснение
  • SLA на tool — держать под 2 секунды, иначе клиент бросит трубку

07 Consent & Escalations

Operational workflows for real deployments. Согласие на запись, передача к менеджеру, SIP на живого оператора.

Consent-задача

Отдельный подкласс AgentTask[bool] с двумя tool-ами: consent given / denied. Задача крутится, пока LLM не получит явное «да» или «нет».

class CollectConsent(AgentTask[bool]):
    def __init__(self, chat_ctx=None):
        super().__init__(
            instructions="""
            Ask for recording consent and get a clear yes or no answer.
            Be polite and professional.
            """,
            chat_ctx=chat_ctx,
        )

    async def on_enter(self) -> None:
        await self.session.generate_reply(instructions=(
            "Briefly introduce yourself, then ask for permission to record "
            "the call for quality assurance and training purposes. "
            "Make it clear that they can decline."
        ))

    @function_tool()
    async def consent_given(self) -> None:
        """Use this when the user gives consent to record."""
        self.complete(True)

    @function_tool()
    async def consent_denied(self) -> None:
        """Use this when the user denies consent to record."""
        self.complete(False)

Эскалация к менеджеру

Отдельный класс ManagerAgent — другой голос, другие полномочия. Передача сохраняет контекст через chat_ctx.

@function_tool()
async def escalate_to_manager(self, context: RunContext) -> ManagerAgent:
    """Transfer the customer to a manager when requested
    or when you cannot resolve their issue."""
    return ManagerAgent(chat_ctx=self.chat_ctx), "Transferring you to a manager now."

Передача живому оператору по SIP

@function_tool()
async def transfer_to_human(self, context: RunContext) -> None:
    """Transfer the call to a human agent."""
    context.disallow_interruptions()
    await context.session.say("I'm transferring you to a human agent now. Please hold.")
    room = get_job_context().room
    await room.local_participant.publish_sip_participant(
        sip_trunk_id="your-trunk-id",
        dial_to="sip:agent@example.com",
    )

Рекомендованный prod-flow

  1. CollectConsent → булев исход; если False, стартуем сессию с record=False
  2. Triage agent → маршрутизация в specialty (тех.поддержка / биллинг / …)
  3. Specialty agent → работает по конкретному сценарию
  4. Manager escalation → если запрошено или не решается
  5. Human handoff по SIP → как последний рубеж

На каждом переходе передавать chat_ctx — иначе новый агент задаёт клиенту те же вопросы заново, что бесит.

Триггеры эскалации

LLM решает вызывать escalate_to_manager из двух источников: docstring tool-а + instructions агента. Практика — прописывать в instructions явные примеры: «If the customer asks to speak with a human, if they say ‘this is ridiculous’, if you tried the same solution twice — escalate».

Что перепроверить перед пилотом

  • Русский STT/TTS — AssemblyAI universal-streaming русский пока не потянет (или потянет с оговорками). Bench: Yandex SpeechKit + Deepgram Nova-2 + собственный Whisper large-v3-turbo на реальном корпусе колл-центра.
  • Русский semantic turn detection — MultilingualModel заявлена, но качество на RU надо тестить на реальных сэмплах.
  • Fallback timeouts / retry counts — не задокументированы, копать код или задавать явно в конфиге.
  • Self-hosted LiveKit Server — для регулируемого контура обязательно; проверить лицензию, ресурсные требования, интеграцию с корпоративным auth (Keycloak/AD).
  • Хорошая практика: любой POC — с baseline (Fitbit-VAD или простой RMS-VAD) + метрики TTFA/interruption_rate + commit hash.
